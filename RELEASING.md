# Releasing

Releases publish to npm via GitHub Actions on a `v*` tag push. There are no
publish credentials to manage — the workflow uses npm's trusted-publisher OIDC
flow.

## Release a new version

1. Bump `version` in `package.json` (semver). Run `npm install --package-lock-only`
   so `package-lock.json` picks up the new version.
2. Open a PR with the bump, get it reviewed, merge to `master`.
3. Pull `master` and tag the merge commit:

   ```sh
   git checkout master && git pull
   git tag -a v0.1.5 -m "Release 0.1.5"
   git push origin v0.1.5
   ```

4. The tag push triggers `.github/workflows/publish.yml`. Watch it complete:

   ```sh
   gh run watch --repo WizaCo/n8n-nodes-wiza
   ```

5. Confirm on npm:

   ```sh
   npm view @wizaco/n8n-nodes-wiza version
   npm view @wizaco/n8n-nodes-wiza dist.attestations
   ```

   The package page on npmjs.com should show a "Provenance" section linking
   to the GitHub Actions run.

## How publishing actually works

- The workflow runs `lint`, `build`, `test`, then `npm publish --provenance --access public`.
- It has `id-token: write` permission so it can request an OIDC token from
  GitHub. npm verifies that token's `repo` and `workflow` claims against the
  package's trusted-publisher config (set via npmjs.com → package settings →
  Trusted Publishers). Match → npm issues a short-lived publish token for
  that single publish.
- The provenance attestation is generated by npm: a SLSA v1 statement signed
  via Sigstore, recorded in the public transparency log, and attached to the
  package. n8n's community-node verification requires this as of May 2026.

## Troubleshooting

**`npm publish` fails with a 401/403** — the npm trusted-publisher entry
doesn't match the workflow. Check that `Repository`, `Workflow filename`, and
the org name still match. If the workflow filename was renamed, update the
trusted-publisher entry.

**`version already exists`** — npm doesn't allow republishing a version. Bump
to the next patch and re-tag. Don't try to delete and re-publish.

**Workflow ran but didn't fire on the tag** — ensure the tag matches `v*`
(e.g. `v0.1.5`, not `0.1.5`). The tag must point to a commit on `master`.

**Provenance section missing on npm page** — check the workflow log for the
"Signed provenance statement" notice. If missing, the `id-token: write`
permission was likely stripped from the workflow.

## Don't do this

- **Don't `npm publish` from a local machine.** The package's 2FA setting
  blocks token-based publishes, so it would fail anyway, but the bigger issue
  is that local publishes don't carry a provenance attestation, which n8n
  cloud verification now requires.
- **Don't add an `overrides` field to `package.json`.** n8n cloud rejects
  community nodes that use it. If a transitive dep needs a security bump,
  raise it upstream (usually `n8n-workflow`) or suppress the finding in our
  vulnerability scanner with a note.
- **Don't force-push tags.** If a publish run fails before `npm publish`
  completes, just bump to the next version and re-tag. Force-pushing a tag
  that already pointed to a published version is a no-op for npm anyway —
  the version is immutable there.
